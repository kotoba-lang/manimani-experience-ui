# Experience rules

Normative language: MUST / MUST NOT / SHOULD / MAY.

1. UI MUST expose context, current state, next action, and evidence.
2. UI MUST distinguish visibility, capability, and sensitivity.
3. A run MUST NOT be called done before a verifiable external receipt exists.
4. A blocker MUST show cause, owner, release condition, and next action.
5. Mascot motion MUST NOT be the only carrier of state.
6. User input MUST receive visible acknowledgement within 100ms.
7. Press and shake MAY add bounded visual impulse; they MUST NOT fake progress or alter authority.
8. Reduced-motion users MUST receive an equivalent static state representation.
9. Destructive and external write actions MUST show actor, scope, target, and preview before execution unless an explicit policy already authorizes them.
10. Rewards MUST follow verified value, not token consumption or compulsive repetition.
11. The UI MUST remain operable without sound, haptics, color discrimination, or mascot animation.
12. The design MUST NOT imitate Nintendo characters, sounds, layouts, or trade dress.
